Project Management: Intended Development Process

$Date: 2002/10/30 20:47:00 $ $Author: istr $ Owner: $Author: istr $ Status: Initial

Details

The development process for the WebScarab project will be a simple, typical open source process where individuals work on different components of interest as needed towards a common goal. Since this model is characterized by disconnected, sometimes chaotic, activity, there will need to be a set of processes in place to help everyone communicate the progress of the project and its direction. To facilitate this, there are certain things we can do:

  • Use a set of workproducts that quickly document the design of the project and other administrivia. This is important both as a record of events, but also as a means of communicating what needs to be done and what has been done.
  • Use a set of clear testing methodologies (such as Unit tests) that everyone can run from their own computers and use to verify code.
  • Have a mailing list (with archives) for the project which allows us to share ideas and stay on top of status.
  • Utilize task lists to determine the path of development.

Since many people are very anxious to write code, and not so enthusiastic to write workproducts, we need to develop a technique to make them not such a chore. I believe the best methodology is the following (to be revised as needed):

  • Start with anything you like: Unit Tests, Working Code, Workproduct/Docs, or even raw design ideas. It should be something that is needed by the project, but not necessarily.
  • Get your initial stab at the design to a point that it is sharable and announce it on the list. Have others look at it and comment.
  • Incorporate improvements into the design, and write another piece of the puzzle: Unit Tests, Working Code, or Workproducts.
  • Throw away anything that is not working.
  • Continue to iterate over each of the three work items (Unit Tests, Working Code, Workproducts), releasing them for review.
  • When they are complete enough, begin to integrate them into the project. Depending on the type of component, this may need to be done during development.
  • If all integration works, and everyone else's Unit Tests are working properly, then you can release it in alpha stage. This allows others to verify their work against yours.
  • After alpha stage for your component, then it should be in the project and will be maintained as usual. Remember to continually revisit your component (or those of others) as changes are made and your Unit Tests break.

Notice that these steps are in a kind or order, but don't necessarily have to be done in that order. People work differently, so whatver works for you is fine. The only requirement is that three things are produced for each thing you work on: Working Code, Unit Tests, Workproducts/Documentation. With these three things, it is very easy for everyone to stay in communication with eachother.
